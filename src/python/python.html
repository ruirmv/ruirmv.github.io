Best Practices

Typing

Use type hints

We should use type hints whenever possible to add clarity to the code and avoid mistakes with a linter.

Imports

Import entire modules instead of individual symbols within a module

  • Reduces cluter
  • Avoids circular imports
Do
import canteen import canteen.sessions from canteen import sessions
Don't
from canteen import get_user from canteen.sessions import get_session

Imports withing a script cannot be relative

Always use absolute imports within a script.

The imports should be in three sections, each separated by a blank line

  1. System imports
  2. Third-party imports
  3. Local source tree imports

All exports from a module must be done from the __init__ file

Public exports belong in the __init__ file.

Classes and Data

Define default values in dictionaries with .get() and .setdefault()

Use data classes

  • require less code since they provide methods like __repr__ and __eq__ out of the box
  • less error prone since they require type hinting
  • can easily be converted to a dictionary or tuple

Class constants

class Constants: VALUE = "value"

Avoid getter and setter methods

Do
class Person: def __init__(self, age): self._age = age @property def age(self): return self._age @age.setter def age_setter(self, age): self._age = age ... person.age = 42
Don't
person.set_age(42)

Styling

Don't use redundant names

An Animal class shouldn't have a method called animal_to_string but instead to_string.

Style

  • Follow PEP 8 when sensible
  • The naming should be representative of the purpose
  • Variables, functions, methods, packages and modules should be named like lower_case_with_underscores
  • Classes and exceptions like CapWords
  • Protected methods and internal functions like _single_leading_underscore(self, ...)
  • Constants like ALL_CAPS_WITH_UNDERSCORES

Prefer code redability

Do
def is_stop_sign(sign): return sign.color = 'red' and sign.sides = 8
Don't
# if the sign is a stop sign if sign.color = 'red' and sign.sides = 8: stop()

Use parenthesis for line continuations

Avoid lines with more than 80-100 characteres.

wiki = ( "Here is a lot of text" "all in the same line" )

Documentation

Documentation should only be provided at the top level, to describe the intent behind a function or a class. If we need to document something on a statement level, probably the code needs to be refactored.

We should follow PEP 257.

Use one-line docstrings for obvious functions:

"""Returns the parameters of foo."""

For more complex functions, multiline docstrings should include

  • Summary line
  • Use case, if appropriate
  • Args
  • Return type and semantics, unless None is returned

Use Sphinx to automatically generate documentation in HTML

Working with time

The package zoneinfo is the prefered way to get timezones

Use UTC by default

import zoneinfo dt = datetime.datetime.utcnow() dt.astimezone("Europe/Lisbon") dt = datetime.datetime(2022, 12, 1, tzinfo=zoneinfo.ZoneInfo("Europe/Lisbon"))
